# 他のツール群のなかでの autopilot の位置

すでに ralph-loop(ループ・プリミティブ)、impeccable(単一スキル OSS テンプレ)、または gstack(開発スイスアーミー)を使っているかもしれません。このページは正直に書きます — autopilot が何を加え、何が重なり、**いつ使わないべきか**。

## ひと目で

| ツール | 主目的 | 起動 | 領域 | 候補自動発掘? | 自己修復? |
|---|---|---|---|---|---|
| **autopilot** | 領域横断の自律走行ミッション | スラッシュ一発 | 何でも | ✅ ミッション・ツール経由 | ✅ Phase 0.5 |
| **ralph-loop** | 汎用ループ・プリミティブ(プロンプトを N 回) | スラッシュ一発 | 何でも | ❌ 毎回自分で書く | ❌ |
| **impeccable** | フロントエンド設計監査 + 修正 | スラッシュ一発 | UI / デザイン | ✅ 設計領域内 | ❌ |
| **gstack** | CLI ツール群(browse、ship、qa、retro…) | インストール + N コマンド | 開発フロー | コマンド毎に部分的 | 該当なし |

## どれをいつ

- **一文で書ける領域で自走するループが欲しい** → `autopilot`
- **プロンプトを N 回だけ繰り返したい、ガバナンス不要** → `ralph-loop`
- **フロントエンド / LP / UX を磨きたい** → `impeccable`(`mission="design polish"` で autopilot も可だが、impeccable は専用設計)
- **開発ショートカット集(browse、qa、ship、retro)** → `gstack`

これらは **排他的ではありません**。autopilot から impeccable サイクルを回すこと、autopilot ミッション内で gstack を呼ぶこと、autopilot が大げさすぎる素朴ループを ralph-loop に任せること、すべて自然です。

## ralph-loop に対して autopilot が足すもの

- **ミッション単位のガバナンス**: 許可パス、禁止領域、リスク階層、diff 上限 — 全 phase で強制
- **自己ペース調整**: 次の起床を cadence + idle バックオフで自動決定。`--max-iterations` 不要
- **自己修復**: 途中クラッシュ、起床ミス、スケジュール紛失を検出して復旧(Phase 0.5)
- **マイルストーン痕跡**: 重要な瞬間を不変ディレクトリへ自動昇格
- **更新通知**: 24h キャッシュで最新リリース確認 + オプトイン自動更新

ralph-loop はプリミティブ、autopilot はその上に手で組まなくてよい形を載せたもの。

## autopilot **ではない** もの

- **コーディング・エージェントではない** — すでに使っているエージェント(Claude Code / Codex / Cursor / Gemini)をオーケストレートするだけ。自前モデルなし。
- **CI の代替ではない** — autopilot が書くのは commit、パイプラインではない。テストは CI が回す。
- **コードスタイルに意見なし** — 読むのは `mission.md` のみ。スタイルはリポと下のエージェントが既に強制している通り。
- **分散オーケストレーターではない** — ディレクトリ毎に単一ミッション。マルチリポは cron + `/schedule`。

## **使わない方がいい** ケース

- **一回限りのタスク** — オーバーヘッドが見合わない。エージェントに直接頼む方が速い。
- **ハードリアルタイム / 本番クリティカルな書き込み** — 既定 L2(小さい PR、main は絶対直接触らない)。本番運用は専用 runbook を。
- **一文に落ちないミッション** — Q1 が書けないなら未成熟。まず brainstorm、次に autopilot。
- **ループにエージェントを任せる信頼がまだない** — autopilot はエージェントの判断をそのまま引き継ぐ。10 サイクル任せられないなら L1(リードオンリー提案)から。

## 60 秒の適合チェック

1. ミッションを一文で書けるか?
2. 許可パスの glob は、PR 毎の人間レビューなしでエージェントが変更しても良い範囲か?
3. verify コマンドは CI がすでに回しているもの(build / test / lint)か?

3 つとも yes → autopilot がフィットする形。
